## 第2页 | 问题背景：为什么需要 RAG

**场景**: 海军标准文档数百份，每份几十页，专业术语密集

**传统方式的痛点**:
- Ctrl+F 关键词匹配 → 查不全（"灭火"搜不到"消防"）
- 人工翻阅比对 → 一份标准几十页，半天找一条条款
- 大模型直接问 → 会编造不存在的标准编号，幻觉严重
- 扫描版 PDF → 文字无法搜索

**RAG 的价值**: 让大模型"先查资料再回答"——检索真实文档 + 基于文档生成答案

## 第3页 | 什么是 RAG Agent

```
RAG = Retrieve（检索）+ Augment（增强）+ Generate（生成）

                      ┌──────────────┐
  用户提问 ──────────→│  Agent 核心   │
                      └──────┬───────┘
                             │
        ① Retrieve          │
        ┌───────────────────┴───────────────────┐
        │  问题 → 向量化 → 语义+关键词混合检索    │
        │  从海量标准中召回最相关的 Top-K 片段    │
        └───────────────────┬───────────────────┘
                             │
        ② Augment           │
        ┌───────────────────┴───────────────────┐
        │  System Prompt = 系统指令 + K个文档片段 │
        │  "你是海军标准专家，基于以下文档回答..." │
        └───────────────────┬───────────────────┘
                             │
        ③ Generate          │
        ┌───────────────────┴───────────────────┐
        │  LLM 流式生成答案 + 强制标注引用来源    │
        │  "根据 [HJB 590B-2025] 第3章第2条..."  │
        └───────────────────────────────────────┘
```

**核心思想**: 不是让大模型"记住"所有标准，而是让它"会查"标准

## 第4页 | ① Retrieve — 混合检索（重点）

**双通道融合，互补长短**

| 通道 | 技术 | 擅长 | 不足 |
|------|------|------|------|
| 语义检索 | BGE嵌入 + pgvector HNSW | "灭火"≈"消防" | 编号"590B"匹配弱 |
| 关键词检索 | PostgreSQL ILIKE | 精确"HJB 590B-2025" | 不懂同义词 |

**RRF 融合算法**:
```
score = 0.7/(60+rank_semantic) + 0.3/(60+rank_keyword)
```

**文档状态加权**: 现行有效×1.0 > 修订中×0.8 > 废止×0.5

**检索流程**:
1. 用户问题 → BGE-base-zh-v1.5 → 768维向量
2. pgvector HNSW 余弦相似度搜索（毫秒级）
3. PostgreSQL ILIKE 关键词模糊匹配
4. RRF 融合排序 → 返回 Top-5 最相关片段

## 第5页 | ② Augment — 上下文增强

**System Prompt 设计（核心工程技巧）**:

```
你是一名海军标准专家，严格基于提供的标准文档片段回答。

【核心原则】
1. 严格依托文档内容作答，严禁编造
2. 无依据则回复"暂无相关标准依据"
3. 标注引用来源：标准编号 + 条款条目

【上下文文档片段】
[文档1] HJB 590B-2025 第3章 第12条
舰艇消防系统应配备...（实际检索到的内容）

[文档2] GJB 4072B-2023 第5章 第8条
军用软件质量监督要求...（实际检索到的内容）

【用户问题】
舰艇消防系统的设计要求和验收标准是什么？
```

**提示词工程要点**:
- 明确角色（海军标准专家）
- 硬约束（严禁编造、必须溯源）
- 结构化输出（区分强制执行/推荐/指导性条款）
- 多轮对话保留最近3轮历史

## 第6页 | ③ Generate — 流式生成

**生成流程**:
```
Context(检索片段) + System Prompt + User Question
        │
        ▼
   通义千问 qwen-plus (兼容 OpenAI API)
        │
        ▼
   SSE 流式输出（逐字显示，类似 ChatGPT/豆包）
        │
        ▼
   后处理：提取引用 → [HJB 590B-2025] 第3章第2节 第12条
```

**防止幻觉的三道防线**:
1. Prompt 硬约束："严禁编造，无依据就说不清楚"
2. 上下文限制：只给检索到的真实文档片段
3. 引用溯源：每条回答必须标注来源，可回查原文

## 第8页 | 文档处理流水线

```
上传 PDF/DOCX/TXT
    │
    ├── 文字型 ──→ pdfplumber 提取文字
    │
    ├── 扫描版 ──→ 自动检测(平均<30字/页)
    │              └→ Tesseract OCR(150DPI + 4线程并行)
    ▼
清洗降噪 ──→ 去页眉页脚/页码/水印
    │
    ▼
元数据提取 ──→ 标准编号(HJB 590B-2025)、名称、日期、归口单位
    │
    ▼
章节分块 ──→ 按"第X章/第X节/第X条"切分，800字/块，100字重叠
    │
    ▼
向量化入库 ──→ BGE嵌入(768维) + execute_values批量写入pgvector
```

## 第9页 | 关键技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 向量数据库 | pgvector (HNSW) | 与业务库一体，免运维，生产可用 |

| OCR | Tesseract + 中文包 | 免费离线，winget 一键安装 |
